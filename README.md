# Redrob AI Hackathon 2026 — Candidate Ranking System

## Directory Structure

```
redrob-ranking/
├── rank.py             # Main pipeline: load -> features -> score -> CSV
├── streamlit_app.py    # Demo UI (upload jsonl, view top 100 + breakdown)
├── requirements.txt
├── README.md
├── candidates.jsonl    # (input, not included)
└── submission.csv      # (output, generated by rank.py)
```

## Running

```bash
pip install -r requirements.txt
python rank.py --input candidates.jsonl --output submission.csv --debug debug.csv
```

`debug.csv` is optional and contains the full per-feature breakdown for the
Top 100 — useful for tuning weights and writing the writeup.

Streamlit demo:

```bash
streamlit run streamlit_app.py
```

## Architecture

1. **Single streaming pass** over `candidates.jsonl` (constant memory,
   regardless of file size). For each candidate:
   - Extract experience (stated vs. derived from career history).
   - Build a "skill text" blob from headline, skills, certifications, and
     career-history titles/descriptions.
   - Compute ~10 interpretable feature scores (see below).
   - Run honeypot heuristics.
2. **Batch semantic similarity**: all skill-text blobs are hashed with a
   `HashingVectorizer` (2^16 dims, 1-2 grams, no training/download needed)
   and compared via sparse cosine similarity to a hand-written JD embedding
   text. This captures *meaning*, not just keyword overlap — directly
   addressing the hackathon's "not the most AI keywords" trap.
3. **Composite scoring** = weighted sum of normalized features, with
   multiplicative penalties for off-target titles, consulting-only
   backgrounds, and honeypot signals.
4. **Honeypot filtering**: candidates with `honeypot_score >= 0.6`
   (2+ independent red flags) are removed from the eligible pool entirely
   before ranking, keeping Top 100 honeypot rate near 0%.
5. **Top 100 selection + monotonic score adjustment** to satisfy the strict
   "monotonically decreasing" requirement even when raw scores tie.
6. **Reasoning generation**: each row picks from several sentence templates
   (seeded per-candidate for determinism) and fills in *that candidate's*
   actual matched skills, title, company, experience, and location — so
   output is varied and grounded in real extracted facts, not boilerplate.

## Feature Engineering

| Feature | Weight | What it measures |
|---|---|---|
| `semantic_sim` | 0.22 | Cosine similarity of candidate text to JD (hashed n-grams) |
| `required_skill` | 0.18 | Coverage of core retrieval/ranking/vector-search stack (capped at 8 matches → diminishing returns) |
| `experience_fit` | 0.16 | Gaussian centered on 6–8 yrs, plateaued for 5–9 yrs acceptable band |
| `eval_skill` | 0.08 | NDCG/MAP/MRR/A-B-testing/offline-eval familiarity |
| `preferred_skill` | 0.07 | LoRA/QLoRA/PEFT/LTR/XGBoost/HRTech/marketplace/OSS |
| `production_signal` | 0.07 | "production", "deployed", "shipped", "scale", etc. near skill mentions |
| `title_score` | 0.09 | Current title relevance (AI/ML/search/ranking engineer vs. negative titles) |
| `company_score` | 0.05 | Penalizes consulting-only career (TCS/Infosys/Wipro/...) unless mixed with product cos |
| `location_score` | 0.04 | Pune/Noida = best, other Tier-1 + open-to-relocate = good, else lower |
| `redrob_score` | 0.04 | open_to_work + recruiter response rate + activity |

**Why diminishing returns on skills?** The hackathon explicitly warns that
keyword-stuffed profiles are a trap. Capping `required_skill` at 8 matches
(score = min(matches,8)/8) and giving `semantic_sim` + `experience_fit` more
combined weight (0.38) than raw skill keyword counts (0.18) means a
profile that genuinely *talks like* a search/ranking/recsys engineer with
the right seniority outranks one that just lists every vector-DB name.

## Honeypot Detection (no external data needed)

Each candidate gets a `honeypot_score` in `{0, 0.3, 0.6, 0.9, 1.0}` based on
how many of these internal-consistency checks fire:

1. **Stated vs. derived experience mismatch** — profile claims N years but
   the career-history date spans only support N-4 or fewer. This is the
   direct analogue of "8 years experience at a company founded 3 years ago."
2. **Expert-level skills with near-zero experience** — 2+ "expert/advanced"
   skill tags, or 8+ skills listed, with <1.5 years total experience.
3. **Inconsistent career history** — end date before start date, or 3+
   years of overlap between unrelated roles.
4. **Experience exceeds education timeline** — claimed experience is more
   than 1.5 years longer than time since (earliest) graduation.
5. **Implausible title jump** — starts as intern/junior/fresher and reaches
   VP/Director/Head/Chief within 2 years.
6. **Company-age mismatch** *(if `redrob_signals.current_company_founded_year`
   is present)* — tenure at current company exceeds the company's age.

`honeypot_score = min(1.0, 0.3 × num_flags)`. Candidates scoring `>= 0.6`
(2+ flags) are dropped from the eligible pool before ranking, and the score
also multiplies surviving candidates' composite score by `(1 - 0.6 ×
honeypot_score)` as a soft penalty for borderline cases.

## Negative Signal Handling

- If the current title matches a negative-title term (Marketing Manager,
  HR Manager, Graphic Designer, Accountant, Content Writer, etc.) **and**
  there's no meaningful production-ML or required-skill signal, the
  composite score is multiplied by 0.25.
- Consulting-only career history (every company in
  TCS/Infosys/Wipro/Cognizant/Capgemini/Accenture) caps `company_score` at
  0.2; mixed consulting+product caps it at 0.7; pure product/startup
  experience gets 1.0.

## Suggestions to Maximize NDCG@10

1. **Tune the weight table against held-out gold rankings** if any are
   provided/synthesizable — even 10–20 hand-labeled "ideal" candidates let
   you grid-search `WEIGHTS` for the combination that best reorders your own
   Top 50 to match.
2. **Expand `REQUIRED_SKILLS` / `PREFERRED_SKILLS` synonyms** — e.g. add
   "ann search", "approximate nearest neighbor", "two-tower", "cross
   encoder", "reranker" — these often appear in real resumes for this exact
   role and currently aren't captured.
3. **Increase `n_features` in `HashingVectorizer`** (e.g. 2^18) if memory
   allows — reduces hash collisions and sharpens semantic similarity, which
   is the single highest-weighted feature.
4. **Add a small interaction term**: `semantic_sim × required_skill` — JD
   alignment *and* concrete tooling overlap together are a stronger signal
   than either alone, and this rewards candidates who are strong on both.
5. **Penalize generic LangChain-only profiles**: add a check — if
   "langchain" appears but none of FAISS/Pinecone/Weaviate/Qdrant/Milvus/
   OpenSearch/Elasticsearch do, apply a small penalty (addresses the "only
   know LangChain demos" negative signal explicitly).
6. **Recency of activity**: if `redrob_signals` includes a last-active date,
   weight recently-active candidates higher — directly maps to "recently
   active engineers" in the JD's strong-candidate profile.
7. **Sanity-check on a small labeled subset first**: run `rank.py
   --debug debug.csv` on ~500 rows, manually eyeball the Top 20 and Bottom
   20, and iterate on `WEIGHTS` / keyword lists before running on the full
   500MB file.
